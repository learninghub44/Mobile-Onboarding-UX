## Backend Platform

Build CHAMA-HUB X on **Supabase** as the primary backend platform.

The backend must use:

* Supabase PostgreSQL
* Supabase Authentication
* Supabase Row Level Security (RLS)
* Supabase Realtime
* Supabase Storage
* Supabase Edge Functions
* Supabase Database Functions
* Supabase Cron (where appropriate)
* Supabase Vector (future AI features)

The architecture should be cloud-first, secure, scalable, and designed to support thousands of organizations.

---

# Authentication

Use Supabase Authentication.

Support:

* Email & Password
* Google Sign-In
* Apple Sign-In (iOS)
* Password Reset
* Email Verification
* Session Persistence
* Multi-device Login
* Secure Session Refresh

Every authenticated user must have a profile linked to their account.

---

# Database

Use PostgreSQL as the single source of truth.

Design normalized tables with proper foreign keys, indexes, constraints, triggers, and audit fields.

Never duplicate data unnecessarily.

Every important table should include:

* id (UUID)
* created_at
* updated_at
* created_by
* organization_id
* status
* deleted_at (soft delete where appropriate)

---

# Multi-Tenant Architecture

The application must support multiple independent organizations.

Every record belongs to exactly one organization.

Users may belong to multiple organizations.

Switching organizations should be seamless.

No organization should ever access another organization's data.

Tenant isolation must be enforced using Supabase Row Level Security (RLS), not only by the client application.

# Artificial Intelligence (Groq) Integration

## Vision

Artificial Intelligence should be a first-class feature of CHAMA-HUB X, enhancing productivity, reducing administrative workload, and helping group leaders make informed decisions. AI must assist users while preserving privacy and never make financial decisions automatically.

The application will use **Groq** as the primary AI inference provider because of its high performance, low latency, and cost efficiency. The AI layer must be designed with an abstraction so that additional providers (such as OpenAI, Anthropic, or Google Gemini) can be added in the future without changing the application architecture.

---

# AI Architecture

Implement an AI service layer with provider abstraction.

```text
Mobile App
      │
AI Service Layer
      │
Groq API
      │
Large Language Model
```

All AI features must communicate through this centralized service. Never call the Groq API directly from UI components.

## AI Safety

The AI must never:

* Modify financial records
* Approve loans
* Delete records
* Change member roles
* Override permissions
* Execute administrative actions
* Fabricate data
* Reveal confidential information

AI should only analyze, summarize, explain, search, and draft content. Any action that changes data must require explicit user confirmation and pass normal authorization checks.

---

## Privacy & Security

Before sending any information to Groq:

* Respect the user's role and permissions.
* Only include the minimum data required for the request.
* Remove unnecessary personally identifiable information where possible.
* Never expose API keys in the client application. All Groq API requests must be proxied through a secure backend or serverless function.
